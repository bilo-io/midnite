# Phase 72 — SSO go-live & operator config split

> [Phase 70](phase-70-google-github-sso.md) built the whole "Continue with Google / GitHub" flow end-to-end — a self-hosted [`SsoService`](../packages/gateway/src/auth/sso.service.ts) issuing the same JWTs `POST /auth/login` does, always-visible [`<SsoButtons>`](../packages/web/components/auth/sso-buttons.tsx), a `__midnite_rt` httpOnly cookie via the web BFF, and a `gateway.auth.sso` config block that references secrets by **env-var name** (`clientSecretEnv`), never inline ([Phase 70 E](phase-70-google-github-sso.md)). [Phase 33](phase-33-multi-user-teams.md) is the JWT base underneath it. **What's missing is not code — it's that SSO has never actually been turned on**, and the config that *would* turn it on lives in the wrong place. The `sso` block (client **IDs**, `redirectUri`, `webBaseUrl`), the `jwt` block, and the `allowlist` all sit under `gateway.auth` in the repo-root [`midnite.json`](../midnite.json) — which is the **user's per-project preferences file** (`agent`, `terminal`, `repos`), **committed to git**. So today an end user can read the operator's Google/GitHub client IDs, and there is no private home for the operator's auth wiring. On top of that, the web app builds with `output: 'export'` **unconditionally** ([`next.config.mjs`](../packages/web/next.config.mjs)), which **silently drops the `/api/auth/*` BFF POST handlers** the SSO cookie flow depends on — so a hosted static build can't complete a sign-in even when configured. **Phase 72 makes SSO real:** carve the whole `gateway.auth` subtree into a **gitignored, operator-owned config source** (fail-closed if it leaks back into `midnite.json`), add a **server web-build target** so the auth routes run hosted, wire the **two real OAuth apps** with explicit per-env redirect URIs, plug the one client-facing config leak, and ship the **turnkey DX** (sample files, `midnite doctor` readiness, a register-apps-to-go-live runbook) so both local and hosted sign-in work first try.

> **Scope guardrails (CLAUDE.md).** Config stays the contract: **all** config is still validated by the single zod schema in [`shared/src/config.ts`](../packages/shared/src/config.ts) and read only through [`loadConfig()`](../packages/shared/src/config-loader.ts) — the operator split changes the *source*, not the typed `MidniteConfig` shape, so the ~10 existing consumers (`sso.service`, `jwt.service`, `auth-policy`, `users.service` allowlist, `health.checkSso`, `enabledSsoProviders`, …) read `config.gateway.auth.*` **unchanged**. Secrets remain **env-var values referenced by name** — the operator file holds client IDs + redirect URIs, never a raw secret. No Firebase, no new provider, no rebuild of the Phase 70 flow or the Phase 33 JWT base. The web server-mode toggle must **default to the existing static export** so desktop packaging is byte-for-byte unchanged. This phase ships a build target + a runbook, **not** provisioned hosting infrastructure.

> Effort tags: **S** small · **M** medium · **L** large. Critical path is **A → B** (operator source + migrate/enforce) then **E** (wire real apps) as the functional core; **D** (server build) unblocks *hosted* sign-in; **C** (leak) + **F** (DX/docs) harden and document. A+B+E is "SSO works locally"; +D is "SSO works hosted".

## Current state (what exists to build on)

- **Config loader** — [`shared/src/config-loader.ts`](../packages/shared/src/config-loader.ts): `loadConfig(explicitPath?)` resolves an **explicit path → else walks up from `cwd()`** to the nearest `midnite.json` (`findConfigPath`, ≤10 parents) → else schema `DEFAULTS`. **No env-var path override exists.** Missing/unparseable → silent defaults. Node-only (kept out of the barrel so the browser never pulls `node:fs`). Gateway injects the parsed `MidniteConfig` via the `MIDNITE_CONFIG` DI token.
- **Auth/SSO schema** — [`shared/src/config.ts`](../packages/shared/src/config.ts): `OAuthClientConfigSchema` (`clientId`, `clientSecretEnv`, `scopes`) → `SsoProviderConfigSchema` (+ optional `redirectUri`) → `GatewaySsoConfigSchema` (`google?`, `github?`, `webBaseUrl?`) → `GatewayAuthConfigSchema` (`tokenEnv`, `requireOnNonLoopback`, `rateLimit`, `jwt{secretEnv,…}`, `sso?`, `allowlist`) → `gateway.auth`. `enabledSsoProviders(config)` returns providers whose `clientId` is present.
- **Consumers of `gateway.auth`** — [`sso.service.ts`](../packages/gateway/src/auth/sso.service.ts) (`resolveClient`/`callbackUri`/`enabledProviders`), [`sso.controller.ts`](../packages/gateway/src/auth/sso.controller.ts) (`webBaseUrl`), [`jwt.service.ts`](../packages/gateway/src/auth/jwt.service.ts) (`secretEnv`/TTLs), [`auth-policy.ts`](../packages/gateway/src/auth/lib/auth-policy.ts) (`tokenEnv`), [`rate-limit.guard.ts`](../packages/gateway/src/auth/rate-limit.guard.ts), [`users.service.ts`](../packages/gateway/src/users/users.service.ts) (`allowlist`), [`gateway-secret-env.ts`](../packages/gateway/src/config/gateway-secret-env.ts) (scrub names), [`bootstrap.ts`](../packages/gateway/src/bootstrap.ts) (`requireOnNonLoopback`), [`health.service.ts`](../packages/gateway/src/health/health.service.ts) (`checkSso`), and shared `enabledSsoProviders`. **A split must keep all of these reading `config.gateway.auth.*` verbatim.**
- **`midnite.json` today** — repo-root, **committed** (`.gitignore` ignores only `.midnite/`, `.env*`, `*.db*`). Holds `agent`/`terminal`/`repos`/`gateway.port`/`workflows` **and** `gateway.auth.allowlist` (two emails). **No `sso`, no `jwt`** block present — SSO is default-absent, so on this checkout JWT is off, providers resolve to `[]`, and buttons fall back to showing both but any click 503s.
- **Env reads** — gateway reads `process.env[...]` directly, keyed by config-supplied names: `jwt.service` `process.env[secretEnv]` (empty ⇒ JWT+SSO disabled), `sso.service.resolveClient` `process.env[clientSecretEnv]`, `auth-policy` `process.env[tokenEnv]`. No `.env.example` exists anywhere in the repo.
- **Web hosting** — [`next.config.mjs`](../packages/web/next.config.mjs) sets `output: 'export'` **unconditionally** (only `ANALYZE` + `NEXT_PUBLIC_APP_VERSION` are env-driven). [`app/api/auth/sso/callback/route.ts`](../packages/web/app/api/auth/sso/callback/route.ts) already documents the intent: it's a POST handler *"so a static export silently skips it … while a real Next server (the hosted, auth-on mode) still runs it"* — but **nothing selects that mode**.
- **Readiness** — [`health.service.ts`](../packages/gateway/src/health/health.service.ts) `checkSso()` already fails-closed on a configured provider with an unset `clientSecretEnv` and warns on "SSO configured but JWT disabled". It runs in `bootChecks()` → the **auth-exempt, rate-limit-exempt** `GET /health/preflight` + `/health/ready` ([`health.controller.ts`](../packages/gateway/src/health/health.controller.ts)), whose `detail`/`remedy` strings **echo provider names + secret env-var names to unauthenticated callers** — the one client-facing config leak.

## Theme A — Operator config source & loader — **M** — ✅ DONE (PR #483, 2026-07-19)

A private, deploy-time home for the operator's auth wiring, deep-merged into the same typed config the whole gateway already reads.

- [x] **`shared`: operator-config resolution + merge** in [`config-loader.ts`](../packages/shared/src/config-loader.ts). A `loadOperatorConfig()` that resolves an operator file **path via `MIDNITE_OPERATOR_CONFIG`**, else a default **`.midnite/operator.json`** (the `.midnite/` dir is *already* gitignored — zero new `.gitignore` work, and it sits beside the runtime data dir). It validates against the operator subtree schema and is **deep-merged into `gateway.auth`** by `loadConfig()` *after* the user file is parsed, so the returned `MidniteConfig` shape is identical to today and every consumer is untouched.
- [x] **Precedence + fail-soft** — operator source is authoritative for `gateway.auth`; a **missing** operator file is fine (SSO simply stays absent, exactly like now), an **unparseable/invalid** operator file **fails closed** (throws — an operator misconfig must never silently fall back to "auth off"). Distinguish "absent" (ok) from "present-but-broken" (error). Preserve the existing "user `midnite.json` missing → defaults" behaviour untouched.
- [x] **Schema seam** — factor the operator-owned subtree into a reusable `OperatorConfigSchema` (wrapping the existing `GatewayAuthConfigSchema`, so no field shapes change) exported from [`config.ts`](../packages/shared/src/config.ts); keep it node-agnostic (the merge/IO lives only in the node-only `config-loader.ts`).
- [x] **Tests (shared)** — resolution precedence (`MIDNITE_OPERATOR_CONFIG` > default `.midnite/operator.json`), deep-merge produces a `MidniteConfig` byte-equal to today's inline form, absent-file no-op, broken-file throws, and the merged `config.gateway.auth.sso` feeds `enabledSsoProviders` unchanged.

## Theme B — Migrate `gateway.auth` out + fail-closed boundary — **M** — ✅ DONE (PR #483, 2026-07-19)

- [x] **Migration ergonomics** — a one-line boot log (info) when an operator config is loaded ("operator auth config loaded: providers=[…], jwt=on/off"), and the fail-closed error names the exact offending key path so an operator can move it in seconds.

## Decisions / open questions

1. **Operator config source** → **Resolved (user): a separate gitignored file, deep-merged.** Default `.midnite/operator.json` (already under the gitignored `.midnite/`), path overridable via `MIDNITE_OPERATOR_CONFIG`. The typed `MidniteConfig` shape and all consumers stay unchanged — only the *source* of `gateway.auth` moves. (Env-only and file+env-override were considered and set aside for a single, structured, reviewable file.)
2. **What moves** → **Resolved (user): the whole `gateway.auth` subtree** (`sso` + `jwt` + `allowlist` + `tokenEnv` + `rateLimit` + `requireOnNonLoopback`). Cleanest boundary — "anything auth is the operator's"; the user file keeps `agent`/`terminal`/`repos`/`gateway.port`. `allowlist` (who may sign in) is operator policy, so it moves too.
3. **Boundary enforcement** → **Resolved (user): fail-closed.** `loadConfig` **errors** if the committed `midnite.json` carries any `gateway.auth` key, with a keyed remedy. Strongest guarantee that client IDs can't be committed by accident; matches the existing fail-closed boot checks.
4. **Web hosting mode** → **Resolved (user): a build-time flag** (`MIDNITE_WEB_TARGET=server`) drops `output:'export'` so the BFF auth routes run hosted; **default stays static** for desktop. One build, two targets.
5. **Health readiness leak** → **Resolved (user): status-only for anonymous probes**, granular `detail`/`remedy` behind auth. Probes keep working; provider + env-var names stop leaking on the auth-exempt routes.
6. **Redirect URIs** → **Resolved (user): explicit per-provider `redirectUri`** in the operator config (schema already supports it) — the registered console value is exactly the configured value; no derivation ambiguity.
7. **DX scope** → **Resolved (user): full.** `operator.example.json` + `.env.example`, `midnite doctor`/preflight SSO readiness, and a register-apps → local + hosted runbook. Turnkey for both environments.
8. **Firebase?** → **Resolved: no** (out of scope, per the seed and Phase 70). The self-hosted `SsoService` already does what Firebase would; this phase only *configures and hosts* it. A Firebase adapter would be a separate future phase and is not planned.
9. **Operator file — auth-only or general?** → **Recommendation: model `OperatorConfigSchema` as a general operator subtree but populate only `gateway.auth` in Phase 72**, so a later phase (e.g. operator-owned rate limits or feature gates) can extend the same private source without another split. *(Recommended; confirm in review.)*
